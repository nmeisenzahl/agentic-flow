# agentic-flow — Contributing Guide

Guide for teams that contribute to the agentic-flow framework itself.

---

## Dog-Food Principle

**All new features to agentic-flow are built using the agentic-flow pipeline itself.**

This means:
1. Open a Feature issue in the `agentic-flow` repository
2. Let the pipeline triage → research → spec → plan → tasks automatically
3. Review and approve artefacts via slash commands
4. Merge the spec PR when all artefacts pass quality review
5. Implement via the generated Task sub-issues

This is the dog-food principle — we trust the pipeline enough to build the pipeline with it.

---

## Running speckit Locally

The pipeline now uses three shipped wrapper agents — `agentic-flow-spec`, `agentic-flow-plan`, and `agentic-flow-tasks` — to adapt speckit to the existing PR/sub-issue environment. For local iteration during development, you can run the underlying speckit agents directly:

```bash
# Generate or update spec.md for an open feature
# (Run from the already-open feature/spec branch for that work)
# In your Copilot Chat or coding session:
# /speckit.specify

# Generate plan.md from an approved spec.md
# /speckit.plan

# Generate tasks.md from an approved plan.md
# /speckit.tasks

# Cross-artifact consistency check
# /speckit.analyze

# Quality checklist evaluation
# /speckit.checklist
```

These are Copilot coding agent sessions — run them in your IDE's Copilot Chat or via the GitHub cloud coding agent.

## Edit / Compile Cycle

All agentic workflows are defined as `.md` source files and compiled to `.lock.yml` files. **Never edit `.lock.yml` directly** — changes will be overwritten the next time the source is compiled.

### Workflow

```bash
# 1. Edit the source file
$EDITOR src/.github/workflows/spec.md

# 2. Compile locally (generates the .lock.yml that GitHub Actions would read in an adopter repo)
gh aw compile src/.github/workflows/spec.md

# 3. Commit the source file (lock files are gitignored in this repo)
git add src/.github/workflows/spec.md
git commit -m "feat(spec): update spec generation workflow"
```

### Compile All

```bash
cd /path/to/agentic-flow
gh aw compile src/.github/workflows/triage.md \
              src/.github/workflows/research.md \
              src/.github/workflows/spec.md \
              src/.github/workflows/plan.md \
              src/.github/workflows/tasks.md \
              src/.github/workflows/refine.md
```

### Compile with Linting

```bash
gh aw compile --actionlint
```

### Verifying Compiled Outputs are Current

The release workflow validates the shipped workflow sources and wrapper agents. To verify your local compile step still produced lock files for smoke-testing:

```bash
for aw in src/.github/workflows/*.md; do
  lock="${aw%.md}.lock.yml"
  [ -f "$lock" ] || echo "MISSING: $lock"
done
```

## Architecture Overview

### `src/` vs Root `.github/`

| Location | Contents | Included in release zip? |
|----------|----------|-------------------------|
| `src/.github/agents/` | Shipped agentic-flow wrapper agents (`agentic-flow-spec`, `agentic-flow-plan`, `agentic-flow-tasks`) | ✅ Yes |
| `src/.github/workflows/` | `.md` agentic workflow sources + standard GHA workflows (`.lock.yml` compiled outputs are gitignored in this repo and generated by adopters) | ✅ Yes |
| `src/.github/workflow-templates/` | Shared template blocks (mcp-github, error-handler, phase-guard) | ✅ Yes |
| `src/.github/copilot/` | `instructions.md` — Copilot context for agents in user repos | ✅ Yes |
| `docs/README.md`, `docs/init.md`, `docs/usage.md` | User-facing docs | ✅ Yes |
| Root `.github/agents/speckit.*.agent.md` | speckit development-only agents | ❌ No |
| Root `.github/workflows/release.yml` | Release packaging workflow (dev tooling only) | ❌ No |
| `docs/contributing.md` | Framework contributor guide (dev tooling only) | ❌ No |
| `.specify/` | Local spec-kit memory layer | ❌ No |
| `specs/` | Feature specs for agentic-flow itself | ❌ No |

### What Goes Where

**`src/.github`** — Only files that adopters need in their repos:
- `src/.github/agents/agentic-flow-spec.agent.md` — wrapper for spec generation/refinement on the assigned branch/PR
- `src/.github/agents/agentic-flow-plan.agent.md` — wrapper for plan generation on the spec PR branch
- `src/.github/agents/agentic-flow-tasks.agent.md` — wrapper for tasks generation on the spec PR branch
- `{phase}.md` workflow sources — adopter repos compile these into `{phase}.lock.yml`
- `triage-trigger.yml` — issues: opened event → dispatch triage agent
- `research-trigger.yml` — label event → dispatch research agent
- `post-merge.yml` — PR merge → create task issues

**Root `.github/`** — Development-only, NOT distributed:
- `speckit.*.agent.md` — speckit phase/gate agents consumed by the wrappers; adopters get these from `specify init`
- `release.yml` — tag push → package and publish release

### Known Constraints

The `issues: opened` event cannot coexist with `slash_command` in a single `.md` file.
This is a current `gh aw` compiler constraint (verified in PA-013-F8). As a workaround,
`triage-trigger.yml` handles the `issues: opened` event and dispatches `triage.lock.yml`
via `workflow_dispatch`. The `/retry-triage` command is handled directly by `triage.md`
(which uses `slash_command: retry-triage` in its frontmatter).
When this constraint lifts in a future `gh aw` release, `triage-trigger.yml` can be removed
and `on: issues: opened` added back to `triage.md`.

## Developing with the Framework

### Edit/Compile Cycle

After editing an `.md` agent file, recompile its lock file:

```bash
gh aw compile src/.github/workflows/<file>.md
```

In this development repo, commit the `.md` source only. Adopter repos compile and commit their own `.lock.yml` files after installation.

### Dogfooding

Using agentic-flow to develop agentic-flow itself (running the framework pipeline
against this repository) is a **planned future capability**. The pipeline workflows
and compiled agents live under `src/.github/`, which GitHub Actions and Copilot do
not scan. A future spec will address how to activate the framework in the dev repo
without duplicating source files.

## Release Process

1. All `src/` and shipped docs changes committed; workflow sources compile locally
2. Tag the release:
   ```bash
   git tag v1.2.3
   git push origin v1.2.3
   ```
3. `release.yml` (at `.github/workflows/release.yml` — not distributed) fires automatically:
   - Validates required files exist (agents, workflows, copilot/instructions.md)
   - Validates shipped wrapper agents, workflow sources, and distributed docs
   - Packages `src/` + `docs/README.md docs/init.md docs/usage.md` into `agentic-flow-v1.2.3.zip`
   - Creates a GitHub Release with auto-generated release notes
   - Appends upgrading notes comparing changed files since previous tag

## Commit Conventions

All commits MUST use [Conventional Commits](https://www.conventionalcommits.org/) format:

```
<type>(<scope>): <description> (#N)
```

Where `#N` is the Feature issue number this commit is associated with.

Examples:
```
feat(triage): add triage agentic workflow (#1)
feat(research): add research agent and trigger workflows (#1)
feat(spec-agent): add agentic-flow spec wrapper agent (#1)
feat(spec): add spec generation workflow (#1)
feat(plan): add plan generation workflow (#1)
feat(tasks): add tasks generation workflow with auto-ready-to-merge (#1)
feat(post-merge): add post-merge task creation workflow (#1)
docs(readme): add consumer docs with prerequisites (#1)
chore(compile): compile all agentic workflows to lock.yml (#1)
chore(release): add release distribution workflow (#1)
```

## Testing

The framework has no compilable source code. Tests are manual acceptance tests:

| Criterion | Test |
|-----------|------|
| SC-001 Triage starts within 5 min | Open test issue; verify the issue is reformatted/labeled within 5 min |
| SC-002 Research in issue ≤15 min | Apply `needs-spec`; time until `## Research Findings` appears |
| SC-003 Full pipeline ≤30 min human time | Run full pipeline; log interaction times |
| SC-004 Zero duplicates | Re-run each command twice; verify no duplicates |
| SC-005 Task issues have working spec links | Verify all blob URLs resolve post-merge |
| SC-006 Human-readable error on failure | Inject failures; verify recovery comments |
| SC-007 Phase ordering enforced | Run commands out of order; verify rejections |
| SC-008 Artefacts pass AI review | Inspect review comments before each approval |
| SC-009 README sufficient for install | Follow README in a fresh repo with no prior knowledge |
| SC-010 Release zip correct structure | Push v0.0.1-test tag; inspect zip contents |
| SC-011 Tasks review rejects non-compliant | Submit tasks.md missing `depends_on`; verify rejection |

Use a dedicated test repository (not a production repo) for acceptance testing.
